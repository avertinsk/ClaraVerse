# Системная архитектура

ClaraVerse — это self-hosted ИИ-workspace с шестью основными сервисами, оркестрированными Docker Compose. Все сервисы работают в одной bridge-сети (`claraverse-network`) и общаются внутренне по имени контейнера.

## Обзор сервисов

| Сервис | Технология | Порт | Роль |
|---------|-----------|------|------|
| **Фронтенд** | React 19 / Vite / Nginx | 80 | Веб-интерфейс, обслуживание статических файлов |
| **Backend** | Go 1.24+ / Fiber | 3001 | REST API, WebSocket, выполнение инструментов |
| **MongoDB** | MongoDB 7 | 27017 | Пользователи, агенты, рабочие процессы, учётные данные, воспоминания |
| **MySQL** | MySQL 8.0 | 3306 | ИИ-провайдеры, модели, возможности, MCP-серверы |
| **Redis** | Redis 7 Alpine | 6379 | Планировщик задач, WebSocket pub/sub, сессии |
| **SearXNG** | SearXNG (latest) | 8080 (внутренний) | Self-hosted мета-поисковая система |

## Поток данных

### Сообщения чата

1. Пользователь отправляет сообщение в браузере.
2. Фронтенд сохраняет сообщение локально в IndexedDB (архитектура нулевого знания).
3. WebSocket-фрейм отправляется на backend на порт 3001.
4. Backend проверяет JWT, проверяет лимиты скорости и пересылает сообщение настроенному LLM-провайдеру.
5. Потоковые фрагменты ответа возвращаются по тому же WebSocket-подключению.
6. Если LLM вызывает инструмент (веб-поиск, выполнение кода и т.д.), backend выполняет его и возвращает результат inline.

### Выполнение рабочего процесса

1. Срабатывает триггер (вручную, по расписанию через Redis cron или API-вызов).
2. Backend загружает определение агента и рабочего процесса из MongoDB.
3. Узлы выполняются последовательно или параллельно, каждый запускает вызовы инструментов или LLM-запросы.
4. История выполнения сохраняется в MongoDB. Отправляются опциональные уведомления вебхуков.

## Что хранит каждая база данных

### MongoDB (документные данные)

| Коллекция | Содержимое |
|------------|----------|
| `users` | Аккаунты и предпочтения |
| `agents` | Определения и конфигурации агентов |
| `workflows` | Определения визуальных рабочих процессов |
| `workflow_versions` | История версий рабочих процессов |
| `executions` | Журналы выполнения агентов |
| `credentials` | Зашифрованные учётные данные интеграций |
| `memories` | Записи системы памяти Clare |
| `chat_sessions` | Опциональная зашифрованная синхронизация чата |
| `analytics` | Аналитика использования |

### MySQL (реляционные данные)

| Таблица | Содержимое |
|-------|----------|
| `providers` | Конфигурации ИИ-провайдеров (OpenAI, Anthropic и т.д.) |
| `models` | Доступные модели для каждого провайдера |
| `model_aliases` | Альтернативные имена моделей |
| `model_capabilities` | Флаги функций для каждой модели |
| `mcp_servers` | Конфигурации MCP-серверов |
| `mcp_server_tools` | Инструменты, предоставляемые MCP-серверами |

### Redis (эфемерные данные)

| Паттерн ключа | Назначение |
|-------------|---------|
| `scheduler:*` | Расписания cron-задач |
| `pubsub:*` | Маршрутизация WebSocket-сообщений между экземплярами |
| `session:*` | Данные сессий |
| `ratelimit:*` | Счётчики ограничения скорости |

## Внешние / опциональные сервисы

| Сервис | Назначение | Обязательно? |
|---------|---------|-----------|
| **SearXNG** | Веб-поиск для агентов | Да (включён в Compose) |
| **E2B** | Песочница выполнения Python-кода | Нет — требует `E2B_API_KEY` |
| **Composio** | OAuth-интеграции (Google Sheets, Gmail и т.д.) | Нет — требует `COMPOSIO_API_KEY` |

## Примечания по масштабированию

- **Backend** горизонтально масштабируем. Redis pub/sub обрабатывает маршрутизацию WebSocket-сообщений между несколькими экземплярами.
- **Фронтенд** обслуживает статические файлы через Nginx и совместим с CDN.
- **Базы данных** следуют стандартным паттернам репликации и шардирования MySQL/MongoDB.
- Для односерверного развёртывания (типичный сценарий self-hosting) лимиты ресурсов по умолчанию в `docker-compose.yml` хорошо работают для ~50 одновременных пользователей.
