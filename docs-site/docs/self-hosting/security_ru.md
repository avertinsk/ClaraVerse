# Руководство по безопасности

ClaraVerse использует локальную JWT-аутентификацию, шифрование AES-256-GCM и многоуровневый middleware для защиты ваших данных. Эта страница охватывает модель безопасности и лучшие практики для продакшен-развёртываний.

## Аутентификация

ClaraVerse использует локальную JWT-аутентификацию (внешние провайдеры аутентификации не требуются).

### Как это работает

1. **Регистрация** — Пароль хешируется с помощью Argon2id и сохраняется в MongoDB.
2. **Вход** — Пароль проверяется против сохранённого хеша. При успехе backend выдаёт access токен и refresh токен (оба JWT, подписанные HS256).
3. **Аутентифицированные запросы** — Фронтенд отправляет access токен в заголовке `Authorization: Bearer <token>`. Middleware проверяет JWT при каждом запросе.
4. **Обновление токена** — Когда access токен истекает, фронтенд использует refresh токен для получения новой пары.

### Время жизни токенов

| Токен | По умолчанию | Переменная |
|-------|---------|----------|
| Access токен | 15 минут | `JWT_ACCESS_TOKEN_EXPIRY` |
| Refresh токен | 7 дней | `JWT_REFRESH_TOKEN_EXPIRY` |

### Уровни безопасности endpoint-ов

```
Запрос
  --> Проверка CORS
    --> Ограничитель скорости
      --> Публичные endpoint-ы (регистрация, вход, health)
      --> Проверка JWT
        --> Пользовательские endpoint-ы (чат, агенты, рабочие процессы)
        --> Проверка администратора
          --> Административные endpoint-ы (управление пользователями, аналитика)
```

Первый зарегистрированный пользователь автоматически становится администратором. Дополнительные суперадминистраторы могут быть назначены через `SUPERADMIN_USER_IDS`.

## Шифрование

### ENCRYPTION_MASTER_KEY

Это самый критический секрет в вашем развёртывании. Он защищает:

- Учётные данные интеграций, сохранённые в MongoDB
- Опционально синхронизируемые сессии чата
- Любые другие пользовательские данные, отмеченные для шифрования

**Сгенерируйте его один раз и храните безопасно:**

```bash
openssl rand -hex 32
```

:::danger
Если вы потеряете `ENCRYPTION_MASTER_KEY`, все зашифрованные данные станут навсегда невосстановимыми. Сохраняйте резервную копию этого ключа отдельно от резервных копий базы данных.
:::

### Как работает шифрование

1. Единый `ENCRYPTION_MASTER_KEY` устанавливается как переменная окружения.
2. Для каждого пользователя уникальный ключ выводится с помощью HKDF с ID пользователя в качестве соли.
3. Данные шифруются с помощью AES-256-GCM, что обеспечивает как конфиденциальность, так и целостность (аутентифицированное шифрование).

```
ENCRYPTION_MASTER_KEY
        |
        v
  HKDF (ID пользователя как соль)
        |
        v
  Ключ пользователя  -->  AES-256-GCM  -->  Зашифрованные данные
```

Это означает:
- Компрометация данных одного пользователя не раскрывает данные других пользователей.
- Мастер-ключ никогда не появляется в базе данных.
- Каждый зашифрованный блок включает тег аутентификации GCM, поэтому несанкционированные изменения обнаруживаются.

## Конфигурация CORS

Переменная `ALLOWED_ORIGINS` контролирует, какие источники могут делать запросы к backend.

**Разработка (по умолчанию):**
```
ALLOWED_ORIGINS=http://localhost,http://localhost:5173,http://localhost:5174,http://localhost:3000,http://localhost:8080
```

**Продакшен:**
```
ALLOWED_ORIGINS=https://yourdomain.com
```

Перечисляйте только точные источники, которые вам нужны. Не используйте `*` в продакшене.

## Ограничение скорости

Ограничение скорости применяется для каждого IP на уровне middleware. Все значения — запросы в минуту.

| Уровень | По умолчанию | Переменная | Применяется к |
|------|---------|----------|------------|
| Глобальный API | 200/мин | `RATE_LIMIT_GLOBAL_API` | Все маршруты `/api/*` |
| Публичное чтение | 120/мин | `RATE_LIMIT_PUBLIC_READ` | Неаутентифицированные GET endpoint-ы |
| Аутентифицированный | 60/мин | `RATE_LIMIT_AUTHENTICATED` | Запросы вошедших пользователей |
| WebSocket | 20/мин | `RATE_LIMIT_WEBSOCKET` | Новые WebSocket-подключения |
| Прокси изображений | 60/мин | `RATE_LIMIT_IMAGE_PROXY` | Запросы прокси изображений |

Для очистки лимитов скорости для конкретного пользователя:

```bash
docker exec -it claraverse-redis redis-cli DEL "ratelimit:user_abc123"
```

## Лучшие практики для продакшена

### 1. Установите окружение в продакшен

```
ENVIRONMENT=production
```

Это отключает `DEV_API_KEY` и включает логирование на уровне продакшена.

### 2. Сгенерируйте надёжные секреты

```bash
# Ключ шифрования (64 hex символа = 256 бит)
openssl rand -hex 32

# JWT-секрет (128 hex символов = 512 бит)
openssl rand -hex 64
```

Никогда не используйте секреты повторно между окружениями.

### 3. Измените пароли базы данных по умолчанию

```
MYSQL_ROOT_PASSWORD=<надёжный-случайный-пароль>
MYSQL_PASSWORD=<надёжный-случайный-пароль>
```

### 4. Разместите обратный прокси перед сервисами

Используйте Nginx, Caddy или Traefik как обратный прокси для:
- Завершения TLS/SSL
- Добавления заголовков безопасности HTTP
- Предоставления дополнительного уровня ограничения скорости

Пример фрагмента Nginx:

```nginx
server {
    listen 443 ssl;
    server_name yourdomain.com;

    ssl_certificate     /etc/ssl/certs/yourdomain.pem;
    ssl_certificate_key /etc/ssl/private/yourdomain.key;

    location / {
        proxy_pass http://localhost:80;
    }

    location /api/ {
        proxy_pass http://localhost:3001;
    }

    location /ws/ {
        proxy_pass http://localhost:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

### 5. Ограничьте источники CORS

Установите `ALLOWED_ORIGINS` только на ваш продакшен-домен.

### 6. Закройте открытые порты

В продакшене рассмотрите удаление маппингов портов хоста для баз данных в `docker-compose.yml`, чтобы они были доступны только внутри сети Docker:

```yaml
# Удалите эти строки для продакшена:
# ports:
#   - "27017:27017"   # MongoDB
#   - "3306:3306"     # MySQL
#   - "6379:6379"     # Redis
```

### 7. Резервное копирование секретов и данных

- Храните `ENCRYPTION_MASTER_KEY` и `JWT_SECRET` в менеджере секретов или безопасном хранилище.
- Регулярно создавайте резервные копии томов MongoDB и MySQL.
- Тестируйте процесс восстановления из резервных копий.

### 8. Обновляйте образы

```bash
docker compose pull
docker compose up -d --build
```

Регулярно обновляйте базовые образы (`mongo:7`, `mysql:8.0`, `redis:7-alpine`, `searxng/searxng:latest`) для получения патчей безопасности.
