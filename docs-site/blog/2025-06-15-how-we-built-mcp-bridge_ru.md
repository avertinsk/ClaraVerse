# Как мы создали Clara Companion: Self-hosted MCP-мост для удалённого доступа к ИИ-инструментам

MCP (Model Context Protocol) мощный — но только когда он можетReach вашу машину. Мы создали Clara Companion, чтобы решить именно этот пробел: Go CLI, который соединяет локальные MCP-серверы с ClaraVerse через WebSocket, чтобы ваш ИИ-ассистент мог получать доступ к файловой системе, git и инструментам памяти отовсюду. Вот история того, почему мы создали его, как он работает под капотом и как вы можете настроить его менее чем за две минуты.

<!-- truncate -->

## Почему нам понадобился мост Model Context Protocol

Когда Anthropic выпустила MCP в конце 2024 года, мы сразу увидели потенциал. MCP даёт ИИ-моделям структурированный доступ к внешним инструментам — файловым системам, базам данных, контролю версий и многому другому. Но была фундаментальная проблема: **MCP-серверы работают локально, а облачный ИИ живёт удалённо.**

У нас было твёрдое убеждение: MCP должен быть подключён там, где находится машина, но ИИ всё ещё должен иметь возможность строить вещи и расширять свои способности даже во время сессии чата. Локальная среда пользователя — это где происходит реальная работа — файлы, репозитории, работающие сервисы. ИИ долженReach всё это в реальном времени.

Мы начали создавать Clara Companion **15 декабря 2024 года** и выпустили первый релиз **1 января 2025 года**. Мы хотели опередить других на рынке с этим концептом моста, и хотя более широкая экосистема MCP двигалась быстро, основная идея выдержала: лёгкий, постоянный туннель между вашими локальными инструментами и облачной ИИ-платформой.

## Как работает архитектура WebSocket-моста

Clara Companion — это единый Go-бинарник, который делает три вещи:

1. **Запускает локальные MCP-серверы** (файловая система, git, память или любой пользовательский сервер)
2. **Открывает постоянное WebSocket-подключение** к backend ClaraVerse
3. **Регистрирует доступные инструменты** и проксирует запросы на выполнение в реальном времени

Архитектура намеренно проста. Когда вы общаетесь с Clarой в браузере, и она решает прочитать файл или выполнить git-команду, запрос течёт так:

```
Чат браузера --> Backend ClaraVerse --> WebSocket --> Clara Companion --> MCP-сервер --> Ваша машина
```

Мост обрабатывает всю сложность: регистрацию инструментов, heartbeats, обновление токенов, экспоненциальный backoff при отключениях и плавное переподключение с автоматической повторной регистрацией.

Вот основа WebSocket-моста из нашего Go-источника:

```go
// Bridge управляет WebSocket-подключением к backend
type Bridge struct {
    backendURL     string
    authToken      string
    conn           *websocket.Conn
    writeChan      chan Message
    onToolCall     func(ToolCall)
    onReconnect    func()
}

// Сообщения текут как типизированный JSON через сокет
type Message struct {
    Type    string                 `json:"type"`
    Payload map[string]interface{} `json:"payload"`
}
```

Каждый вызов инструмента от backend приходит как сообщение `tool_call`. Companion выполняет его на соответствующем MCP-сервере и отправляет обратно `tool_result`. Канал записи буферизован (100 сообщений), и heartbeats на уровне приложения (30с) и WebSocket-пинги (45с) держат подключение живым через прокси и брандмауэры.

## Почему мы выбрали Go для кросс-платформенных MCP-инструментов

Go был очевидным выбором для CLI, который должен работать везде. Одна команда `go build` создаёт статически скомпилированный бинарник без зависимостей времени выполнения. Мы поставляем готовые бинарники для:

- **macOS** (Intel и Apple Silicon)
- **Linux** (amd64)
- **Windows** (amd64)

Никаких виртуальных окружений Python, никаких менеджеров версий Node.js, никакого Docker на стороне клиента. Скачайте, аутентифицируйтесь, запустите. Бинарник обычно меньше 15 МБ и запускается за миллисекунды.

Модель горутин Go также чисто отображается на потребности параллелизма моста: одна горутина читает из WebSocket, одна пишет, и вызовы инструментов выполняются параллельно без блокировки подключения.

## Установка и настройка Clara Companion

Начало работы занимает три команды:

```bash
# 1. Установка бинарника (macOS / Linux)
curl -fsSL https://raw.githubusercontent.com/ClaraVerse/ClaraVerse/main/backend/mcp-bridge/scripts/install.sh | sh

# 2. Аутентификация с вашим аккаунтом ClaraVerse
clara_companion login

# 3. Запуск интерактивной панели
clara_companion
```

На Windows используйте PowerShell:

```powershell
irm https://raw.githubusercontent.com/ClaraVerse/ClaraVerse/main/backend/mcp-bridge/scripts/install.ps1 | iex
```

Команда `login` использует **OAuth 2.0 Device Authorization Grant** (RFC 8628). Она отображает короткий код в вашем терминале, открывает браузер и ждёт, пока вы авторизуетесь. Никакие пароли не вставляются в CLI, и это работает с любым провайдером аутентификации — email, Google, GitHub.

```
Для аутентификации откройте этот URL в вашем браузере:
  https://claraverse.app/device

Затем введите этот код:

   ┌────────────────┐
   │   ABCD-1234    │
   └────────────────┘

Ожидание авторизации... (истекает через 15:00)
```

После авторизации companion сохраняет учётные данные в `~/.claraverse/mcp-config.yaml` и запускает панель TUI.

## Добавление и управление self-hosted MCP-серверами

Clara Companion поставляется с тремя MCP-серверами по умолчанию, которые покрывают наиболее распространённые случаи использования:

| Сервер       | Что делает                      | Пакет                                        |
|-------------|-----------------------------------|-------------------------------------------------|
| `filesystem` | Чтение, запись и список файлов       | `@modelcontextprotocol/server-filesystem`       |
| `git`        | Статус, diff, коммит, лог         | `@modelcontextprotocol/server-git`              |
| `memory`     | Постоянный граф знаний        | `@modelcontextprotocol/server-memory`           |

Вы можете добавить любой MCP-совместимый сервер одной командой:

```bash
# Добавить сервер автоматизации браузера
clara_companion add browser --command npx --args @browsermcp/mcp@latest

# Добавить пользовательский сервер по пути
clara_companion add mytools --command /usr/local/bin/my-mcp-server

# Список всех настроенных серверов
clara_companion list

# Удалить сервер
clara_companion remove browser
```

Конфигурация живёт в едином YAML-файле:

```yaml
backend_url: wss://claraverse.app/mcp/connect
device:
  device_id: dev_abc123
  refresh_token: eyJ...
  user_email: user@example.com
mcp_servers:
  - name: filesystem
    command: npx
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/home/user"]
    enabled: true
  - name: git
    command: npx
    args: ["-y", "@modelcontextprotocol/server-git"]
    enabled: true
```

## Запуск как фоновый сервис на macOS и Linux

Для постоянного доступа Clara Companion может установить себя как фоновый сервис:

```bash
clara_companion service install
```

На **macOS** это создаёт plist `launchd` по адресу `~/Library/LaunchAgents/com.claraverse.clara-companion.plist`, который автоматически запускает мост при входе в систему и перезапускает его при крахе.

На **Linux** это создаёт пользовательский сервис `systemd` по адресу `~/.config/systemd/user/claraverse-mcp.service` с `Restart=always` и задержкой перезапуска 10 секунд.

```bash
# Проверить, работает ли сервис
clara_companion service status

# Просмотр логов
tail -f ~/.claraverse/logs/clara_companion.log

# Остановка или перезапуск
clara_companion service stop
clara_companion service start
```

Мост обрабатывает прерывания сети плавно. Если WebSocket падает, он переподключается с экспоненциальным backoff (начиная с 1 секунды, максимум 60 секунд) и автоматически перерегистрирует все инструменты. Обновление токенов происходит заранее — за пять минут до истечения — чтобы сессии не прерывались посреди беседы.

## Что мы узнали, создавая ранний MCP-мост

Создание Clara Companion до созревания экосистемы MCP научило нас нескольким вещам:

**Выпускайте самое простое, что работает.** Наша первая версия была просто WebSocket + прокси инструментов. Без TUI, без управления сервисами, без аутентификации устройств. Этот основной цикл — подключиться, зарегистрировать, проксировать, ответить — почти не изменился с декабря 2024 года.

**Go — правильный инструмент для распространения CLI.** Мы рассматривали Node.js (выравнивание экосистемы) и Rust (производительность), но простота сборки Go и модель горутин победили. Кросс-компиляция для шести целей — это один shell-скрипт.

**Аутентификация устройств стоит усилий.** Вставка токенов в терминал — это риск безопасности и плохой UX. Поток OAuth 2.0 Device Grant добавляет несколько сотен строк кода, но делает онбординг отполированным.

Экосистема MCP быстро выросла с тех пор, как мы начали. Но фундаментальная проблема — соединение локальных инструментов с удалённым ИИ — остаётся актуальной независимо от того, self-hosted вы или используете управляемую платформу. Clara Companion даёт вам этот мост с единым бинарником и нулевой инфраструктурой.
