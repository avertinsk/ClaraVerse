# Руководство по безопасности ClaraVerse

Комплексная проверка безопасности и документация для ClaraVerseAI.

## Обзор безопасности

ClaraVerse реализует многоуровневую защиту (defense-in-depth):

```
┌─────────────────────────────────────────────────────────────┐
│                    Уровни безопасности                        │
├─────────────────────────────────────────────────────────────┤
│  1. Сетевой уровень     │ CORS, TLS, Ограничение частоты    │
│  2. Аутентификация      │ JWT (HS256), пароли Argon2id      │
│  3. Авторизация         │ Ролевая, владение ресурсами       │
│  4. Защита данных       │ Шифрование AES-256-GCM, HKDF      │
│  5. Валидация ввода     │ Валидация схем, саниризация       │
│  6. Логирование и аудит │ События безопасности, логи доступа│
└─────────────────────────────────────────────────────────────┘
```

## Аутентификация

### Система JWT-токенов

ClaraVerse v2.0 использует локальную JWT-аутентификацию с подписью HS256.

**Реализация:** `backend/pkg/auth/local_jwt.go`

| Тип токена | Срок действия по умолчанию | Настраивается через |
|------------|---------------------------|---------------------|
| Access Token | 15 минут | `JWT_ACCESS_TOKEN_EXPIRY` |
| Refresh Token | 7 дней | `JWT_REFRESH_TOKEN_EXPIRY` |

**Claims токена:**
```go
type JWTClaims struct {
    UserID  string `json:"sub"`
    Email   string `json:"email"`
    Role    string `json:"role"`
    TokenID string `json:"jti"`  // Для отслеживания refresh-токенов
    jwt.RegisteredClaims
}
```

**Свойства безопасности:**
- Подписано HS256 (HMAC-SHA256)
- Валидация издателя (`claraverse-local`)
- Принудительное истечение срока
- Уникальные ID токенов для refresh-токенов

### Хеширование паролей

**Алгоритм:** Argon2id (рекомендовано OWASP)

**Реализация:** `backend/pkg/auth/local_jwt.go:146-150`

```go
const (
    argon2Time    = 3           // Количество итераций
    argon2Memory  = 64 * 1024   // 64 МБ памяти
    argon2Threads = 4           // Фактор параллелизма
    argon2KeyLen  = 32          // 256-битный ключ
    saltLength    = 16          // 128-битная соль
)
```

**Почему Argon2id:**
- Memory-hard: Устойчив к атакам GPU/ASIC
- Устойчивость к побочным каналам (id вариант)
- Победитель конкурса Password Hashing Competition
- Рекомендованные параметры OWASP

### Middleware аутентификации

**Реализация:** `backend/internal/middleware/auth.go`

**Проверки безопасности:**
1. Извлечение токена (заголовок Authorization или параметр запроса)
2. Проверка подписи
3. Валидация истечения срока
4. Извлечение claims и сохранение в контексте

**Безопасность продакшна:**
```go
// КРИТИЧНО: Никогда не разрешать обход аутентификации в продакшне
if environment == "production" {
    log.Fatal("КРИТИЧЕСКАЯ ОШИБКА БЕЗОПАСНОСТИ: Аутентификация не настроена в продакшне")
}
```

## Авторизация

### Ролевой контроль доступа

**Роли:**
| Роль | Уровень доступа |
|------|-----------------|
| `user` | Стандартные пользовательские эндпоинты |
| `admin` | Панель администратора + все пользовательские эндпоинты |

### Доступ администратора

**Реализация:** `backend/internal/middleware/admin.go`

**Два метода доступа администратора:**

1. **Поле роли (предпочтительно):**
   ```go
   if role == "admin" {
       // Предоставить доступ администратора
   }
   ```

2. **SUPERADMIN_USER_IDS (legacy):**
   ```bash
   SUPERADMIN_USER_IDS=user-id-1,user-id-2
   ```

### Владение ресурсами

Все пользовательские ресурсы ограничены по `user_id`:
- Агенты
- Рабочие процессы
- Учётные данные
- Воспоминания
- Сессии чата
- Загрузки

**Пример проверки владения:**
```go
if file.UserID != userID {
    return c.Status(fiber.StatusForbidden).JSON(fiber.Map{
        "error": "Доступ к этому файлу запрещён",
    })
}
```

## Шифрование данных

### Шифрование в состоянии покоя

**Алгоритм:** AES-256-GCM (Аутентифицированное шифрование)

**Реализация:** `backend/internal/crypto/encryption.go`

**Получение ключа:**
```
ENCRYPTION_MASTER_KEY (32 байта hex)
         │
         ▼
    ┌─────────┐
    │  HKDF   │◀── ID пользователя (соль) + "claraverse-user-key" (info)
    │ SHA-256 │
    └────┬────┘
         │
         ▼
   Ключ для пользователя (32 байта)
         │
         ▼
    ┌─────────┐
    │ AES-256 │◀── Случайный 12-байтовый nonce для каждого шифрования
    │   GCM   │
    └────┬────┘
         │
         ▼
   Зашифрованные данные
   (nonce || ciphertext || auth_tag)
```

**Что шифруется:**
- Учётные данные пользователей (API-ключи, OAuth-токены)
- Данные синхронизации чата (при использовании серверной синхронизации)
- Чувствительные предпочтения

**Свойства безопасности:**
- Изоляция ключей для каждого пользователя
- Тег аутентификации предотвращает подделку
- Случайный nonce предотвращает replay-атаки
- Мастер-ключ никогда не хранится в базе данных

### Хранение в браузере

Переписки хранятся в IndexedDB по умолчанию:
- **Архитектура с нулевым знанием** — сервер никогда не видит содержимое чата
- Шифрование происходит на стороне клиента
- Даже администраторы не могут читать чаты пользователей

## Ограничение частоты

**Реализация:** `backend/internal/middleware/rate_limiter.go`

### Конфигурация ограничений

| Тип эндпоинта | По умолчанию | Переменная окружения |
|---------------|-------------|---------------------|
| Global API | 200/мин | `RATE_LIMIT_GLOBAL_API` |
| Public Read | 120/мин | `RATE_LIMIT_PUBLIC_READ` |
| Authenticated | 60/мин | `RATE_LIMIT_AUTHENTICATED` |
| WebSocket | 20/мин | `RATE_LIMIT_WEBSOCKET` |
| Image Proxy | 60/мин | `RATE_LIMIT_IMAGE_PROXY` |
| Transcribe | 10/мин | Закодировано |
| Upload | 10/мин | Закодировано |

### Ответ при превышении лимита

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json

{
  "error": "Слишком много запросов. Подождите перед повторной попыткой."
}
```

### Генерация ключей

Ограничения частоты отслеживаются по:
- ID пользователя (аутентифицированные запросы)
- IP-адресу (анонимные запросы)

## Конфигурация CORS

**Реализация:** `backend/cmd/server/main.go`

```go
app.Use(cors.New(cors.Config{
    AllowOriginsFunc: func(origin string) bool {
        // Проверка против ALLOWED_ORIGINS
    },
    AllowMethods:     "GET,POST,PUT,DELETE,OPTIONS",
    AllowHeaders:     "Origin,Content-Type,Accept,Authorization",
    AllowCredentials: true,
    MaxAge:           86400,
}))
```

**Конфигурация:**
```bash
ALLOWED_ORIGINS=https://yourdomain.com,https://app.yourdomain.com
```

### Эндпоинты триггеров (открытый CORS)

Внешние эндпоинты триггеров (`/api/trigger/*`) имеют разрешительный CORS:
- Требуется для интеграций с вебхуками
- Защищено аутентификацией API-ключом

```
cv_live_<случайная_строка>
```

## Валидация ввода

### Валидация запросов

Все входные данные API валидируются:
- Валидация JSON-схемы
- Проверка типов
- Ограничения размера
- Обязательные поля

**Пример:**
```go
type CreateAgentRequest struct {
    Name        string   `json:"name" validate:"required,min=1,max=100"`
    Description string   `json:"description" validate:"max=1000"`
    Model       string   `json:"model" validate:"required"`
    Tools       []string `json:"tools" validate:"max=20"`
}
```

### Валидация загрузки файлов

- Ограничения размера файла
- Валидация MIME-типа
- Саниризация имён файлов
- Сканирование на вирусы (если настроено)

## Логирование и аудит

### Логирование событий безопасности

```go
log.Printf("✅ Аутентифицирован пользователь: %s (%s)", user.Email, user.ID)
log.Printf("🚫 Пользователь %s без прав администратора попытался получить доступ к админ-эндпоинту", userID)
log.Printf("⚠️  [ОГРАНИЧЕНИЕ] Достигнут лимит загрузки для пользователя: %v", userID)
log.Printf("🚫 [БЕЗОПАСНОСТЬ] Пользователю %s отказано в доступе к файлу %s", userID, fileID)
```

### Уровни логов

| Уровень | Применение |
|---------|------------|
| INFO | Нормальные операции |
| WARN | События безопасности (ограничения, отказ в доступе) |
| ERROR | Ошибки, исключения |
| FATAL | Критические ошибки безопасности (продакшн без аутентификации) |

## Чек-лист безопасности продакшна

### Обязательная конфигурация

- [ ] **Установите надёжные секреты:**
  ```bash
  ENCRYPTION_MASTER_KEY=$(openssl rand -hex 32)
  JWT_SECRET=$(openssl rand -hex 64)
  ```

- [ ] **Настройте CORS:**
  ```bash
  ALLOWED_ORIGINS=https://yourdomain.com
  ```

- [ ] **Установите окружение:**
  ```bash
  ENVIRONMENT=production
  ```

- [ ] **Настройте TLS** (через обратный прокси)

- [ ] **Измените пароль администратора по умолчанию**

### Рекомендуемая конфигурация

- [ ] Включить ограничение частоты (включено по умолчанию)
- [ ] Настроить агрегацию логов
- [ ] Настроить мониторинг/оповещения
- [ ] Регулярные обновления безопасности
- [ ] Резервные копии баз данных с шифрованием

### Анти-паттерны безопасности, которых следует избегать

| Анти-паттерн | Риск | Смягчение |
|--------------|------|-----------|
| Закодированные секреты | Раскрытие учётных данных | Используйте переменные окружения |
| Режим отладки в продакшне | Утечка информации | Установите `ENVIRONMENT=production` |
| Разрешительный CORS | CSRF-атаки | Белый список конкретных origins |
| Отсутствие ограничений частоты | DoS-атаки | Включите ограничение частоты |
| Логирование чувствительных данных | Раскрытие данных | Санирируйте логи |

## Обработка уязвимостей

### Сообщение о проблемах безопасности

Сообщайте об уязвимостях безопасности через:
- GitHub Security Advisories (приватно)
- Email: security@claraverse.space

### Процесс обновления безопасности

1. Проблема классифицируется в течение 24 часов
2. Разрабатывается и тестируется исправление
3. Координированное раскрытие
4. Публикуется advisory безопасности
5. Пользователи уведомляются

## Соответствие

### Соответствие GDPR

ClaraVerse поддерживает требования GDPR:

| Требование | Реализация |
|------------|------------|
| Экспорт данных | `GET /api/user/data` |
| Удаление данных | `DELETE /api/user/account` |
| Согласие | Принятие политики конфиденциальности |
| Минимизация данных | Хранение в браузере по умолчанию |

### Хранение данных

Настраивается по тарифам:
- Хранение истории чата
- Хранение логов выполнения
- Автоматическая очистка через фоновые задачи
