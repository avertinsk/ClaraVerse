# Руководство по архитектуре ClaraVerse

Этот документ описывает системную архитектуру, компоненты и поток данных ClaraVerseAI.

## Обзор

ClaraVerse — это приватное ИИ-пространство, построенное как монорепозиторий с бэкендом на Go и фронтендом на React. Он предоставляет чат, генерацию изображений и визуальную автоматизацию рабочих процессов, сохраняя конфиденциальность пользователей через хранение данных в браузере.

```
ClaraVerseAI/
├── backend/           # API-сервер Go 1.24+
├── frontend/          # React 19 + Vite + TypeScript
├── searxng/           # Конфигурация self-hosted поиска
└── docker-compose.yml # Оркестрация сервисов
```

## Компоненты

### Фронтенд (React 19)

| Компонент | Описание |
|-----------|-------------|
| **Чат** | Потоковый чат в реальном времени с локальным хранением IndexedDB |
| **Конструктор агентов** | Визуальный редактор рабочих процессов для создания ИИ-агентов |
| **Панель администратора** | Управление пользователями, аналитика, конфигурация провайдеров |
| **Учётные данные** | Безопасное управление учётными данными интеграций |
| **Память** | Просмотр и управление системой памяти Клары |
| **Настройки** | Пользовательские предпочтения и конфигурация API-ключей |

**Ключевые технологии:**
- React 19 с TypeScript
- Vite для сборки
- Tailwind CSS 4 для стилизации
- Zustand для управления состоянием
- IndexedDB для хранения чата с нулевым знанием

### Бэкенд (Go Fiber)

| Слой | Компоненты |
|------|------------|
| **Обработчики** | Обработчики HTTP/WebSocket запросов |
| **Middleware** | Аутентификация, ограничение частоты, CORS, проверки администратора |
| **Сервисы** | Бизнес-логика (Чат, Агент, Рабочий процесс, Память и т.д.) |
| **Модели** | Структуры данных и схемы баз данных |
| **Инструменты** | 80+ встроенных реализаций инструментов |

**Ключевые технологии:**
- Go 1.24+
- Fiber (веб-фреймворк в стиле Express)
- WebSocket для потоковой передачи в реальном времени
- JWT для аутентификации

### Базы данных

#### MySQL (Реляционные данные)

Хранит конфигурации провайдеров и метаданные моделей.

```sql
-- Основные таблицы
providers        -- Конфигурации ИИ-провайдеров (OpenAI, Anthropic и т.д.)
models           -- Доступные модели для каждого провайдера
model_aliases    -- Альтернативные имена моделей
model_capabilities -- Флаги функций для каждой модели

-- MCP таблицы
mcp_servers      -- Конфигурации MCP-серверов
mcp_server_tools -- Инструменты, предоставляемые MCP-серверами
```

#### MongoDB (Документные данные)

Хранит пользовательские данные, агентов и рабочие процессы.

```javascript
// Коллекции
users            // Учётные записи и предпочтения пользователей
agents           // Определения и конфигурации агентов
workflows        // Определения визуальных рабочих процессов
workflow_versions // История версий рабочих процессов
executions       // История выполнений агентов
credentials      // Зашифрованные учётные данные интеграций
memories         // Система памяти Клары
chat_sessions    // Опциональная синхронизация чата (зашифрованная)
analytics        // Аналитика использования
```

#### Redis (Эфемерные данные)

Обрабатывает планирование задач и обмен сообщениями в реальном времени.

```
scheduler:*      // Расписания cron-задач
pubsub:*         // Маршрутизация WebSocket-сообщений
session:*        // Данные сессий
```

### Внешние сервисы

| Сервис | Назначение |
|--------|------------|
| **SearXNG** | Self-hosted мета-поисковая система для веб-поиска |
| **E2B** | Изолированное выполнение кода (локальный Docker или облако) |
| **Composio** | OAuth интеграции (Google Sheets, Gmail) |

## Поток данных

### Поток сообщений чата

```
1. Пользователь отправляет сообщение (Фронтенд)
   └─▶ IndexedDB сохраняет локально
   └─▶ WebSocket отправляет на бэкенд

2. Бэкенд получает сообщение
   └─▶ Проверка JWT
   └─▶ Проверка ограничения частоты
   └─▶ Обработка сервисом чата

3. Потоковый ответ LLM
   └─▶ Обнаружены вызовы инструментов → Выполнение инструментов
   └─▶ Потоковая передача чанков через WebSocket

4. Фронтенд получает поток
   └─▶ Обновление UI в реальном времени
   └─▶ Сохранение в IndexedDB
```

### Поток выполнения рабочего процесса

```
1. Триггер (Ручной/Расписание/API)
   └─▶ Загрузка агента + определение рабочего процесса

2. Выполнение узлов (последовательное/параллельное)
   └─▶ Преобразование входных данных
   └─▶ Выполнение инструмента/LLM
   └─▶ Маппинг выходных данных

3. Результаты
   └─▶ Сохранение истории выполнений
   └─▶ Уведомления через вебхуки (если настроены)
```

## Архитектура аутентификации

ClaraVerse v2.0 использует локальную аутентификацию JWT (переход от Supabase).

```
┌──────────────────────────────────────────────────────────────┐
│                    Поток аутентификации                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Регистрация                                                  │
│  ┌────────┐     ┌──────────┐     ┌──────────┐              │
│  │ Клиент │────▶│ /register│────▶│ Argon2id │              │
│  └────────┘     └──────────┘     │  Хеш     │              │
│                                   └────┬─────┘              │
│                                        ▼                     │
│                                   ┌──────────┐              │
│                                   │ MongoDB  │              │
│                                   │  users   │              │
│                                   └──────────┘              │
│                                                              │
│  Вход                                                         │
│  ┌────────┐     ┌──────────┐     ┌──────────┐              │
│  │ Клиент │────▶│  /login  │────▶│ Проверка │              │
│  └────────┘     └──────────┘     │ пароля   │              │
│       ▲                          └────┬─────┘              │
│       │                               ▼                     │
│       │         ┌──────────────────────────┐               │
│       └─────────│ JWT Access + Refresh     │               │
│                 │ Токены (HS256)           │               │
│                 └──────────────────────────┘               │
│                                                              │
│  Аутентифицированный запрос                                  │
│  ┌────────┐     ┌──────────┐     ┌──────────┐              │
│  │ Клиент │────▶│Middleware│────▶│ Проверка │              │
│  │ + JWT  │     │          │     │   JWT    │              │
│  └────────┘     └──────────┘     └────┬─────┘              │
│                                        ▼                     │
│                                   ┌──────────┐              │
│                                   │Обработчик│              │
│                                   └──────────┘              │
└──────────────────────────────────────────────────────────────┘
```

**Конфигурация токенов:**
- Access Token: 15 минут (настраивается через `JWT_ACCESS_TOKEN_EXPIRY`)
- Refresh Token: 7 дней (настраивается через `JWT_REFRESH_TOKEN_EXPIRY`)

## Архитектура шифрования

Пользовательские данные шифруются с помощью AES-256-GCM с использованием ключей, полученных для каждого пользователя.

```
│                   Поток шифрования                           │

│           │                                                 │
│           ▼                                                 │

│  │      HKDF       │◀── ID пользователя (соль)             │
│  │ Получение ключа │                                       │
│  └────────┬────────┘                                       │
│           │                                                 │
│           ▼                                                 │
│  ┌─────────────────┐     ┌─────────────────┐              │
│  │ Ключ для польз. │────▶│   AES-256-GCM   │              │
│  └─────────────────┘     │   Шифрование    │              │
│                          └────────┬────────┘              │
│                                   │                        │
│                                   ▼                        │
│                          ┌─────────────────┐              │
│                          │ Зашифр. данные  │              │

│                          │  синхр. чата)   │              │

```

## Архитектура инструментов

ClaraVerse включает 80+ встроенных инструментов, организованных по категориям.

### Категории инструментов

| Категория | Инструменты |
|-----------|-------------|
| **Поиск и данные** | Веб-поиск, поиск изображений, веб-скрейпер |
| **Вычисления** | Математика, Python, аналитик данных, ML-тренер |
| **Время** | Текущее время, конвертация часовых поясов |
| **Документы** | Генерация PDF, чтение документов, таблицы |
| **Коммуникации** | Slack, Discord, Telegram, Teams, Email |
| **CRM** | HubSpot, LeadSquared |
| **Управление проектами** | ClickUp, Jira, Linear, Trello, Asana |
| **Разработка** | GitHub, GitLab, тестирование API |
| **E-commerce** | Shopify товары/заказы/клиенты |
| **Аналитика** | Mixpanel, PostHog |
| **Хранилище** | AWS S3, MongoDB, Redis |
| **Google Workspace** | Sheets, Gmail (через Composio) |

### Регистрация инструментов

```go
// Встроенные инструменты регистрируются при запуске
func registerBuiltInTools(r *Registry) {
    r.Register(NewSearchTool())
    r.Register(NewMathTool())
    // ... 80+ инструментов
}

// Инструменты MCP для конкретного пользователя
r.RegisterUserTool(userID, mcpTool)
```

### MCP Bridge

Model Context Protocol (MCP) bridge позволяет подключать внешние инструменты.

```
┌────────────┐     WebSocket      ┌────────────┐
│  Фронтенд  │◀──────────────────▶│  Бэкенд    │
│ MCP Клиент │                    │ MCP Bridge │
└────────────┘                    └─────┬──────┘
                                        │
                                        ▼
                                  ┌────────────┐
                                  │ MCP Сервер │
                                  │  (stdio)   │
                                  └────────────┘
```

## Архитектура развёртывания

### Docker Compose (Рекомендуется)

```yaml
services:
  frontend    # React приложение (порт 80)
  backend     # Go API (порт 3001)
  mongodb     # Документное хранилище (порт 27017)
  mysql       # Реляционное хранилище (порт 3306)
  redis       # Кеш/планировщик (порт 6379)
  searxng     # Поисковая система (внутренний)
  e2b-service # Выполнение кода (внутренний)
```

### Масштабирование

- **Бэкенд**: Горизонтально масштабируемый с Redis pub/sub для маршрутизации WebSocket-сообщений
- **Фронтенд**: Статические ресурсы, CDN-friendly
- **Базы данных**: Стандартные паттерны масштабирования MySQL/MongoDB
- **E2B**: Размер пула настраивается через `E2B_SANDBOX_POOL_SIZE`
